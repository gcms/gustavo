Extending Feature Models with Relative Cardinalities

 

Gustavo Sousa - Walter Rudametkin - Laurence Duchien

Spirals - Université Lille 1 / CRIStAL / Inria Lille Nord Europe

 

Summary

 

This accompanying site contains additional information about the research conducted on the use of feature models for modeling variability in cloud computing environment. Here we make available the code base of the implementation of relative cardinalities into feature models, the experimentation code as well as the database of the results obtained. All the files and paths mentioned here can be found in the bundle zip.

 

Feature modeling tool

 

The domain specific language and editor was developed using Xtext and EMF technology on Eclipse Luna.  The language meta-model was designed using Ecore and the Eclipse support for OCLinEcore. The model allows describing feature models with relative cardinalities and configurations for a feature model. The meta-model allows for defining complete or partial configurations. A sample feature model and partial configuration are shown below.

Feature models for the analyzed cloud providers are available in src/main/resources/.

 

The Eclipse workspace containing the meta-model, grammar and code used to implement the DSL can be found in the eclipse-workspace subdirectory. The generated plugin files, which can be used to install the modeling tool in Eclipse can be found in eclipse-workspace/plugins/.

To load the workspace or use the editor plugin you need to download Eclipse Luna and install the following software into it:

-       Xtext Complete SDK

-       OCL Classic SDK: Ecore/UML Parsers,Evaluator,Edit

-       OCL Examples and Editors SDK

 

 

Feature model analysis tool

 

The feature model analysis tool is mostly coded in Groovy with some small portions in Java. It uses the developed Xtext plugin for parsing feature models and Choco Solver to solve the Constraint Satisfaction Problem (CSP). From a users point of view the main classes are XtextLoader, SolverBuilder and FeatureModelSolver. A sample usage of these classes can be seen below, and many other examples can be found in the SolverBuilderTests class.


        URL fileURL = new File("src/main/resources/OpenShift.fm");
        FeatureModel fm = XtextLoader.INSTANCE.load(fileURL.toURI().toURL());

        // Build a solver for the feature model
        SolverBuilder builder = new SolverBuilder();
        FeatureModelSolver solver = builder.setFeatureModel(fm).buildSolver();

        // Find a configuration for the feature model
        Iterator configIterator = solver.solutions.iterator();
        if (!configIterator.hasNext())
            return; // No solution found

        Configuration config = configIterator.next();

        // Build a solver to verify a configuration
        FeatureModelSolver checker = builder.setConfiguration(config.toConfig()).buildChecker()

        // Check the configuration
        assertTrue checker.isSatisfied()

        // Load config from file
        File configFile = new File("src/main/resources/configOpenShift.fm");
        Configuration configFromFile = XtextLoader.INSTANCE.loadConfiguration(configFile.toURI().toURL());
        checker = builder.setConfiguration(configFromFile).buildChecker()

        assertTrue checker.isSatisfied()

 

 

The tool can be easily built using Gradle, all needed dependencies are either automatically fetched or are included in the bundle. The config command line script is a wrapper for the fr.inria.spirals.fm.solver.Main class that lets you generate configurations from a feature model file or verify the compliance of a configuration to a feature model.

 

Evaluation

 

The evaluation experiments are implemented by the classes in the package fr.inria.spirals.fm.eval, which includes classes for generating random feature models, adding relative cardinalities and measuring the execution time of operations. Most classes can be parameterized, so that different feature models can be generated or loaded from other sources. By default the parametrization used for generation is the one described in the paper.

A set of command line tools is also provided to enable full customization of this process. The following executable scripts are provided for convenience to execute the corresponding classes.

-       fma (FeatureModelStatsMain) extracts information from feature models such as the number of features with cardinalities and relative cardinalities, the number of constraints and constraining expressions, etc. Can be provided a feature model file or a directory with multiple feature models.

-       generate (FeatureModelGeneratorMain) generates random feature models. A set of properties can be customized such as the number of features, maximum depth, maximum cardinality, probability of having a cardinality etc.

-       add_relative_cardinality (RelativeCardinalityGeneratorMain) adds relative cardinalities to a feature model. Process feature models to randomly add relative cardinalities. The feature models can be loaded from the database, files or standard input (pipe with generate).

-       eval (fr.inria.spirals.fm.eval.Main) is the main evaluation script, used for most experiments in the paper. Has many command line options for configuring the source of feature models to be processed, which operations should be performed and where results should be stored.

 

All feature models and measurements of the execution are stored by default in a SQLite database which is available in the src/main/resources/eval/eval.db.  Together with the database we have the queries and data obtained for the results shown in the paper in the same directory, as well as the employed plotting scripts.
